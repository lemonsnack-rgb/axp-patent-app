# AXP 선행기술검색 프로토타입 — 기능정의서

문서 목적: 이번에 만든 **목업/프로토타입(웹에서 실제 렌더되는 기능 기준)**의 기능을 페이지별로 정의하고, 각 화면 요소가 **어느 API에서 로드되어야 하는지**를 매핑한다.

원칙(작성 규칙):
- **API 명세에만 있고 UI에 없는 항목의 누락은 허용**한다(문서에서 생략 가능).
- **UI(웹 렌더)에 있는데 이 문서에서 누락되면 안 된다.** → UI 요소는 전부 기술하고, 대응 API가 없으면 `⚠ API 미정의(목업)`로 명시한다.

## 0. 자료 출처

| 구분 | 출처 | 성격 |
|---|---|---|
| A. 저수준 API 매뉴얼 | 로컬 `검색 API명세서/[JARVIS-PATENT-API] _search_bigtext(-group-by, -string-list-group-by)` | ISE BIGTEXT wrapper. snake_case, `data{}` 봉투 |
| B. UI/UX 요구사항 | 로컬 `검색 API명세서/[AXP] 특허검색 동작방식 - 개발노트` | 화면 동작 요구사항([검색-NN]) + 표시필드 매핑 |
| C. 실제 프런트/BFF 계약 | `github.com/muhayu/ck-patent`(private, `dev`) `application/patent-front/src/entities/prior-art/*`, `entities/enum/model/*SearchField.ts` | 실제 UI가 호출하는 BFF 와이어 계약(camelCase) |
| D. 프로토타입 | 본 저장소(`src/views/*`, `src/components/PatentDetail.tsx` 등) | 이번에 만든 목업(웹 렌더 기준) |

> 참고: 사용자가 지정한 `github.com/muhayu/prior-art-search`는 현재 접근 권한으로 404. 동일 기능의 실제 구현·계약은 `muhayu/ck-patent`의 `patent-front`에 존재하여 이를 기준(C)으로 삼았다.

## 1. 명세 비교 및 차이 (분석)

### 1.1 API는 2계층 구조다

```
[UI/프로토타입]
    │  호출
    ▼
(BFF) /resources/search/prior-art/{muhayu-patents | muhayu-papers | patent-facets | paper-facets}   ← 자료 C (실제 UI가 쓰는 계약)
    │  래핑
    ▼
(ISE) /search/bigtext , /search/bigtext-group-by , /search/bigtext-string-list-group-by            ← 자료 A (로컬 PDF 매뉴얼)
```

- **자료 A(로컬 PDF)** 는 최하위 ISE 검색엔진 wrapper를 문서화한 것이다. 필터(국가·상태·기간)를 별도 파라미터로 받지 않고 `search_query`/`select_query`/`filter_query`에 접어 넣는다. 응답도 `field_name[]`+`result[][]` 원시 배열이다.
- **자료 C(실제 BFF)** 는 UI 친화 계약이다. 필터를 **구조화 파라미터**(statuses/docTypes/countries/ipcMain/cpcMain/dates/years/venues)로 받고, 응답을 **타입드 객체**(`Patent`/`Paper`) + `columns`(60+ 컬럼 패스스루)로 돌려준다. 패싯도 `FacetAxisResult`로 정규화한다.
- 프로토타입은 자료 C 계약과 **거의 일치**한다(관측된 실제 호출 `POST /search/prior-art/muhayu-patents {searchQuery, fetchOffset, fetchCount, orderByQuery, statuses, docTypes, countries}` 와 동일 형태).

### 1.2 두 명세 간 주요 차이

| 항목 | 자료 A (ISE bigtext) | 자료 C (BFF, 실제) |
|---|---|---|
| 엔드포인트 | `/search/bigtext(-group-by…)` | `/resources/search/prior-art/{muhayu-patents/papers, patent/paper-facets}` |
| 표기 | snake_case, `data{}` 봉투 | camelCase, 봉투 없음 |
| 검색식 | `search_query` (필터 포함해 직접 작성) | `searchQuery` + 구조화 필터 파라미터 분리 |
| 필터(특허) | select/filter query에 포함 | `statuses[] docTypes[] countries[] ipcMain[] cpcMain[] dateFrom/To applicationYears[]` |
| 필터(논문) | 동일하게 query 포함 | `venues[] years[] yearFrom/To` |
| 응답 | `field_name[]`+`result[][]` 원시 | `hits[].patent/paper`(타입드) + `columns{}` 패스스루 |
| 패싯 | group-by/string-list-group-by(gsize/gkey) | `facets.{축}: {items[{key,count}], totalGroups, fetched, error}` |
| 스니펫 하이라이트 | `keyword_char_offset` 제공 | BFF hit엔 미노출(프런트는 클라이언트 하이라이트) |

### 1.3 요구사항(B) ↔ 실제 응답(C) 불일치 — 확인 필요

| 내용 | B(동작방식) | C(실제 BFF) | 판정 |
|---|---|---|---|
| 특허 패싯 축 | 국가·**권리상태**·IPC주분류·CPC주분류·출원연도 (`검색-150`) | `countryCode, ipcMain, cpcMain, applicationYear` | ⚠ **권리상태(register_status) 패싯이 실제 응답에 없음** |
| 논문 국가 | 국가 개념 없음(`검색-211`) | `Paper.countryCode: null` | 일치(항상 null) |

### 1.4 프로토타입 UI에는 있으나 API(A·C)에 정의가 없는 항목 (⚠ 목업 전용 — 연동 시 정의 필요)

아래는 **웹에 렌더되지만 대응 API 필드가 없는** 요소다. 규칙상 문서에서 누락하면 안 되므로 전부 명시하고, 각 페이지 매핑표에서 `⚠ API 미정의(목업)`로 표기한다.

- **특허 상세**: 도면(figures)·대표도면·부호의 설명(refSigns), 패밀리정보, 인용·피인용 목록, 상세설명의 구조화 소제목(기술분야/배경기술/해결과제/해결수단/효과/도면의설명/구체적내용), 청구항 목록 구조(독립/종속), 심판/소송 정보, 특허평가 등급, 출원인/발명자/대리인 **주소**, 출원인 식별코드, 우선권주장일·심사청구일·존속기간만료일·권리변동.
  - 이 중 청구항(`claim`)·상세설명(`specification`)·IPC/CPC 리스트는 자료 A `display` 필드에 존재 → BFF `columns` 패스스루로 수급 가능. 도면·패밀리·인용·심판/평가·주소류는 **A·C 모두 미정의**.
- **논문 결과/상세**: 권(volume)·호(issue)·시작/끝 페이지·발행 월·논문유형(thesis)·학위수여기관. (`Paper` 타입엔 title/authors/journal/publicationYear/publicationDate/keywords/abstract/citationCount/internal·externalFileUrl/score 만 존재)
- **공통**: 라이브러리 저장, 결과 CSV(클라이언트 생성), 검색 히스토리(로컬 상태), 특허↔논문 교차검색, 명세서 이월 — 모두 프런트 상태/클라이언트 기능(백엔드 API 불필요).

## 2. BFF 계약 요약 (자료 C — 매핑 기준)

**공통**: `POST /resources/search/prior-art/…`, `Content-Type: application/json`, 실패 시 `{message}`.

### 2.1 특허 검색 `/muhayu-patents`
- 요청 `PatentSearchRequest`: `searchQuery`, `fetchOffset?`, `fetchCount?`(기본 20), `orderByQuery?`, `statuses?[]`, `docTypes?[]`, `dateFrom?`, `dateTo?`, `countries?[]`, `ipcMain?[]`, `cpcMain?[]`, `applicationYears?[]`
- 응답 `PatentSearchResponse`: `totalCount`, `fetchCount`, `hits[]`(`PatentHit{ patent, dbfile, rowid }`)
- `Patent`: `docid, inventionTitle, inventionTitleEng, abstract, applicant, inventorName, agentName, applicationNumber, applicationDate, publicationDate, registerDate, openDate, searchBaseDate, ipcMain, cpcMain, registerStatus, countryCode, documentKind, columns{60+}`

### 2.2 논문 검색 `/muhayu-papers`
- 요청 `PaperSearchRequest`: `searchQuery`, `fetchOffset?`, `fetchCount?`, `orderByQuery?`, `yearFrom?`, `yearTo?`, `venues?[]`, `years?[]`
- 응답 `PaperSearchResponse`: `totalCount`, `fetchCount`, `hits[]`(`PaperHit{ paper, dbfile, rowid }`)
- `Paper`: `paperId, title, titleEng, abstract, authors[], journal, publicationYear, publicationDate, keywords[], internalFileUrl, externalFileUrl, citationCount, countryCode(null), score`

### 2.3 패싯 `/patent-facets`, `/paper-facets`
- 요청: 검색요청과 동일 필터(검색어+필터).
- 특허 응답 `facets: { countryCode, ipcMain, cpcMain, applicationYear }` (각 `FacetAxisResult{ items[{key,count}], totalGroups, fetched, error }`)
- 논문 응답 `facets: { journal, publicationYear }`

### 2.4 검색식 표기(자료 B·프로토타입 공통)
- 필드절 `CODE:(값)`, 일자 `CODE:([from ~ to])`, 범위코드 `KEY`(명칭+요약+독립항)/`TAC`(명칭+요약+전체청구항)/`DSC`(명칭+요약+전체청구항+상세설명). 자유어는 선두에만 범위 적용(`검색-21`), 사용자 직접 필드식은 불변(`검색-22`).
- 필드코드 카탈로그는 자료 C `patentSearchField.ts`(TI/AB/CL/CLI/CLA/KEY/TAC/DSC/…/IPCM/CPCM/AP/INV/AG/AN/PN/RN/AD/PD/RD 등 70+), `paperSearchField.ts`.

## 3. 페이지별 기능명세 (8개)

표기: 요소 → **[API]** 소스. `client`=프런트 계산/상태, `⚠목업`=API 미정의.

### 3.1 특허 · 검색어 입력 (`PatentInput`)

목적: 변리사가 검색식·필터를 구성해 특허 검색을 실행한다. (요구사항 §2)

주요 기능
- 메타필터: **국가**(전체/KR/US/JP/CN/EP/PCT/DE/GB + 추가국가) · **문헌종류**(전체/공개/등록/실용공개/실용등록) · **기간**(전체/최근5·10·20년 + 일자범위) · **상태**(전체/A:출원·심사중·등록 / I:거절·소멸·취하·포기·무효). 활성 칩은 체크(✓)+색상. (`검색-40~42`)
- 검색어 범위 탭: 명칭+요약+독립항 / 명칭+요약+전체청구항 / 명칭+요약+전체청구항+상세설명 (`검색-20`)
- 입력방식: 일반모드(textarea) / 편집기모드(구문강조+실시간 진단, 실행 차단 안 함) (`검색-11·12`)
- 검색식 입력창 + 우측 검색/초기화 (`검색-10·50·53`)
- 항목별 상세검색(필드별 조합, 펼침/접힘, 필드 추가·확장 모달) → 값 입력 시 검색식에 AND 누적(`검색-60~63`), 실행 후 입력값만 비움·구성 유지(`검색-70~72`)
- 검색 히스토리(검색식·검색일시·★즐겨찾기·재검색, 최대 100, 10개↑ 페이지네이션) — `client`(로컬 store)

화면요소 → API 매핑

| 요소 | 소스 |
|---|---|
| 검색 실행 payload `searchQuery` | 검색식(입력창) + 항목별필드 AND 누적 + 범위코드 래핑 → `PatentSearchRequest.searchQuery` |
| 국가 칩 | → `countries[]` |
| 문헌종류 칩 | → `docTypes[]` |
| 기간(프리셋/일자) | → `dateFrom`/`dateTo` |
| 상태(A/I 칩) | → `statuses[]` |
| 정렬 기본 | → `orderByQuery`(예 `application_date DESC`, 기본 `$SCORE DESC`) |
| 필드코드 목록/타입 | 자료 C `patentSearchField.ts`(text/date-range/ipc) |
| 검색 히스토리·즐겨찾기 | `client` |
| 키워드 추천/파인더 | `⚠목업`(현재 비노출) |

### 3.2 특허 · 결과 목록 (`PatentResults` / `TableResults`)

목적: 검색 결과를 표/카드로 보여주고 좁히기·정렬·페이지 이동·선택 저장. (요구사항 §4·§5)

주요 기능
- 상단 요약바: 적용 검색식 칩 · **총 건수** · → 논문으로(교차검색)
- 데스크톱 표 / 모바일 카드. 컬럼: 체크박스·상태·문헌번호·원문(PDF)·발명의명칭(+새 탭)·출원인·출원일·IPC (`검색-90`)
- 정렬(출원일/공개일/등록일 + 방향), perPage, CSV 다운로드
- 페이지네이션(새 검색 시 1페이지, `검색-120~122`)
- 패싯 사이드바/필터바: 국가·현재권리상태·IPC·CPC·출원연도, 체크 후 실행(`검색-150~155`)
- 체크박스 다중 → 라이브러리 일괄 저장 · 로딩/0건/오류 상태(`검색-140~142`)
- 행 클릭 → 오버레이(3.3) 열기

화면요소 → API 매핑

| 요소 | 소스 |
|---|---|
| 총 건수 | `PatentSearchResponse.totalCount` |
| 행 목록 | `hits[].patent` |
| 상태 배지 | `patent.registerStatus` |
| 문헌번호 | `patent.countryCode`+공백+번호(우선순위 등록>공고>공개>출원: `registerNumber`>`publicationNumber`>`openNumber`>`applicationNumber`, `columns`에서)+공백+`documentKind` (자료 B 규칙) |
| 발명의 명칭 | `patent.inventionTitle`(+`inventionTitleEng`) |
| 출원인 | `patent.applicant` |
| 출원일 | `patent.applicationDate` |
| IPC | `patent.ipcMain` |
| 원문 PDF 아이콘 | `⚠목업`(원문 URL 필드 미정의; `columns` 내 원문키 확인 필요) |
| 명칭 키워드 하이라이트 | `client`(요구 `검색-100`; BFF엔 offset 미노출) |
| 정렬/페이지/perPage | → `orderByQuery`/`fetchOffset`/`fetchCount` |
| 필터칩·패싯 건수 | `/patent-facets` `facets.{countryCode,ipcMain,cpcMain,applicationYear}` (⚠ 권리상태 패싯은 응답에 없음 → 상태는 검색요청 `statuses`로만) |
| CSV | `client`(hits→CSV) |
| 체크 저장 | `client`(라이브러리) |

### 3.3 특허 · 오버레이 상세 (우측 드로어 · `PatentDetail embedded`)

목적: 목록을 유지한 채 우측에서 문서 상세를 빠르게 확인. (요구사항 §6, 좁은 화면=슬라이드오버)

주요 기능
- 헤더: 이전/다음·위치(n/총) · 새 탭 열기 · 라이브러리 저장 · 닫기
- 키워드 하이라이터 바
- 제목 블록(상태·국가·문헌번호·평가등급 + 원문 PDF 다운로드) → 앵커 탭(서지/인명/요약/상세설명/청구범위/패밀리/인용·피인용/분류/기타) → 스크롤 본문 + 도면(본문 내)

화면요소 → API 매핑 (상세는 3.4와 동일 소스, 단일 컬럼 렌더)

| 요소 | 소스 |
|---|---|
| 이전/다음·위치 | `client`(목록 인덱스) |
| 새 탭/저장/닫기 | `client` |
| 상세 본문 전체 | 3.4 매핑 참조 |

### 3.4 특허 · 상세 페이지 (새 탭 · `StandaloneDetail`→`PatentDetail full`)

목적: 문서 전체 정보를 문서형(중앙정렬·2단)으로 정독. 좌: 제목+탭+섹션, 우: 도면 rail, 하단 푸터.

화면요소 → API 매핑

| 섹션/요소 | 소스 |
|---|---|
| 상태 배지 | `patent.registerStatus` |
| 국가/문헌번호/문헌종류 | `patent.countryCode`/번호규칙/`documentKind`(→docKind 파생) |
| 발명의 명칭 | `patent.inventionTitle`(+Eng) |
| 원문 PDF 다운로드 | `⚠목업`(원문 URL 미정의) |
| **서지사항** 문헌일 | `patent.searchBaseDate` |
| 〃 출원번호/출원일 | `patent.applicationNumber`/`applicationDate` |
| 〃 공개번호/공개일 | `patent.openDate` + 공개번호(`columns`) |
| 〃 공고번호/공고일 | `patent.publicationDate` + 공고번호(`columns`) |
| 〃 등록번호/등록일 | `patent.registerDate` + 등록번호(`columns`) |
| 〃 권리상태 | `patent.registerStatus` |
| 〃 우선권주장일·심사청구일·존속만료일·권리변동 | `⚠목업`(A·C 미정의; 만료일은 등록상태 시 출원일+20년 규칙-자료 B) |
| 〃 타임라인 | 위 일자들로 `client` 조합 |
| **인명정보** 출원인 | `patent.applicant` |
| 〃 발명자 | `patent.inventorName` |
| 〃 출원인/발명자 주소·식별코드 | `⚠목업`(미정의; `columns`에 존재 시 매핑) |
| **요약** | `patent.abstract` |
| **상세설명**(본문) | `columns.specification`/`big_specification` (자료 A `display` 필드) |
| 〃 기술분야/배경기술/과제/해결수단/효과/도면설명/구체적내용 소제목 | `⚠목업`(구조 분해는 미정의 — 원문은 specification) |
| **청구범위**(독립/전체) | `columns.claim` (자료 A). 독립/종속 구조 분해는 `⚠목업` |
| **패밀리 정보** | `⚠목업`(미정의) |
| **인용·피인용** | `⚠목업`(enum BCC/FCC 존재하나 hit/columns 노출 미확인) |
| **분류코드** IPC/CPC | `patent.ipcMain`/`cpcMain` (전체 리스트는 `columns.ipc_number_list`/`cpc_code_list`) |
| **기타** 대리인 | `patent.agentName` |
| 〃 대리인주소/심판·소송/특허평가 등급 | `⚠목업`(미정의) |
| **도면**(대표/그 외/부호의 설명/확대뷰) | `⚠목업`(도면 API 미정의) |
| 푸터 | `client`(정적) |

### 3.5 논문 · 검색어 입력 (`PaperInput`)

목적: 논문 검색식·필터 구성·실행. 흐름은 특허와 대칭(`검색-213`).

주요 기능
- 발행연도 필터(전체/최근1·5·10년 + 연도 from~to) (`검색-211`, 국가 없음)
- 검색어 범위 탭: 제목+초록 / 제목+초록+전문
- 검색식 입력창 + 검색/초기화
- 항목별 상세검색(제목·초록·키워드·전문·저자 등, 펼침/접힘) → AND 누적
- 검색 히스토리(특허와 동일) — `client`

화면요소 → API 매핑

| 요소 | 소스 |
|---|---|
| 검색 payload | → `PaperSearchRequest.searchQuery` (범위코드 래핑 + 필드 AND 누적) |
| 발행연도(프리셋/from~to) | → `yearFrom`/`yearTo` (또는 `years[]`) |
| 정렬 | → `orderByQuery` |
| 필드코드 | 자료 C `paperSearchField.ts`(제목·초록·키워드·전문·저자) |

### 3.6 논문 · 결과 목록 (`PaperResults`)

목적: 논문 결과 카드 목록·좁히기·정렬·페이지·선택 저장.

주요 기능
- 요약바: 검색식 칩·총 건수·→ 특허로(교차검색)
- 카드: 제목(+대등제목) · 메타라인(저자, 저널 권(호), 페이지, 발행연도(월)) · **초록**(앞에 "초록" 라벨) · 새 탭 열기 · 체크박스
- 필터바/패싯: 저널·발행연도
- 정렬·perPage·CSV(제목·저자·저널·발행연도) · 페이지네이션 · 다중선택 저장
- 카드 클릭 → 오버레이(3.7)

화면요소 → API 매핑

| 요소 | 소스 |
|---|---|
| 총 건수 | `PaperSearchResponse.totalCount` |
| 카드 목록 | `hits[].paper` |
| 제목/대등제목 | `paper.title`/`paper.titleEng` |
| 저자 | `paper.authors[]` |
| 저널명 | `paper.journal` |
| 발행연도 | `paper.publicationYear` (발행일 `publicationDate`) |
| 권(호)·시작/끝페이지·발행 월·학위수여기관 | `⚠목업`(Paper 타입 미정의) |
| 초록(+"초록" 라벨) | `paper.abstract` |
| 초록 하이라이트 | `client` |
| 필터/패싯 건수 | `/paper-facets` `facets.{journal, publicationYear}`; 요청 `venues[]`/`years[]` |
| CSV/체크 저장 | `client` |

## 4. 종합 — 연동 시 반드시 정의가 필요한 항목(우선순위)

1. **특허 도면 API**(도면 이미지·대표도면·부호의 설명) — 상세/오버레이 핵심인데 A·C 모두 미정의.
2. **특허 원문 PDF URL** — 목록/상세의 원문 버튼.
3. **특허 서지 보강 필드**(공개/공고/등록 번호, 우선권주장일·심사청구일·존속만료일·권리변동, 주소류) — 다수는 `columns` 패스스루로 수급 가능한지 확인 필요.
4. **특허 패밀리·인용/피인용** — 상세 섹션.
5. **특허 권리상태 패싯** — 요구사항엔 있으나 `/patent-facets` 응답에 없음(추가 or 상태는 검색필터로만).
6. **논문 서지 보강**(권/호/페이지/월/논문유형/학위수여기관/분야/영문저널/영문초록) — `Paper` 타입 확장 필요.
7. **상세설명/청구항 구조화**(소제목·독립/종속 분해) — 원문(specification/claim)은 있으나 구조 분해 규칙 정의 필요.

> 위 `⚠목업` 항목은 현재 프로토타입에서 목업 데이터로 렌더된다. UI에는 존재하므로 본 문서에 모두 명시했다(누락 없음). 실제 연동 시 위 목록을 API 계약에 반영하면 된다.
