# AXP 선행기술검색 프로토타입 — 기능정의서 (검색 기능 · 상세 제외)

문서 목적: 이번에 만든 **목업/프로토타입(웹에서 실제 렌더되는 기능 기준)**의 기능을 페이지별로 정의하고, 각 화면 요소가 **어느 API에서 로드되어야 하는지**를 매핑한다.

> ⚠️ **범위(2026-07-06 재조정)**: **특허 수집·서비스 DB가 아직 설계 전**이다. 상세페이지는 수집 필드가 확정돼야 정의할 수 있으므로, 본 문서는 **상세 비의존 기능(검색어 입력·결과 목록·공통)**을 먼저 정의한다. **오버레이 미리보기와 전체 상세페이지는 §5로 보류**한다. (근거: 아래 §0, 상세 필드 후보는 [상세페이지-수집필드-대조.md](상세페이지-수집필드-대조.md)·[상태값 정의서](상세페이지-상태값-정의.md)에 별도 정리)

원칙(작성 규칙):
- **UI에 실제 렌더되는 것만 기재한다(내용 창작 금지).** 코드로 확인되지 않는 동작·필드는 추정해 쓰지 않는다. 미구현·미확정은 그대로 `⚠ 미구현 / 연동 시 정의`로 표기한다.
- **API 명세에만 있고 UI에 없는 항목의 누락은 허용**한다(문서에서 생략 가능).
- **UI(웹 렌더)에 있는데 이 문서에서 누락되면 안 된다.** → In-범위 UI 요소는 전부 기술하고, 대응 API가 없으면 `⚠ API 미정의(목업)`로 명시한다.

> `github.com/muhayu/prior-art-search`는 데모 저장소로, `docs/spec/search-ux.md`가 자료 B(동작방식 PDF)와 **동일 내용**임을 확인했다. 이 데모의 BFF(`/search/prior-art/*`)가 곧 실제 UI 계약이며 자료 C′(ck-patent)와 일치한다.

**상세페이지 보류(§5)**: prior-art 검색 API에는 문서 상세 전용 엔드포인트가 없고, **상세에 노출할 필드는 수집 DB 설계에 좌우**된다(현재 미설계). 상세/오버레이는 목록 검색 응답의 `hit`(+`columns`)를 재사용해 렌더하지만, 어떤 필드를 어떻게 채울지는 DB 확정 후 정의한다. 따라서 상세(오버레이 포함)는 §5에 사유·후속 정의 대상만 남기고 본 §3 명세에서는 제외한다.

## 1. 명세 비교 및 차이 (분석)

### 1.1 API는 2계층 구조다

```
[UI/프로토타입]
    │  호출
    ▼
(BFF) /resources/search/prior-art/{muhayu-patents | muhayu-papers | patent-facets | paper-facets}   ← 자료 C (실제 UI가 쓰는 계약)
    │  래핑
    ▼
(ISE) /search/bigtext , /search/bigtext-group-by , /search/bigtext-string-list-group-by            ← 자료 A (로컬 PDF 매뉴얼)
```

- **자료 A(로컬 PDF)** 는 최하위 ISE 검색엔진 wrapper를 문서화한 것이다. 필터(국가·상태·기간)를 별도 파라미터로 받지 않고 `search_query`/`select_query`/`filter_query`에 접어 넣는다. 응답도 `field_name[]`+`result[][]` 원시 배열이다.
- **자료 C(실제 BFF)** 는 UI 친화 계약이다. 필터를 **구조화 파라미터**(statuses/docTypes/countries/ipcMain/cpcMain/dates/years/venues)로 받고, 응답을 **타입드 객체**(`Patent`/`Paper`) + `columns`(60+ 컬럼 패스스루)로 돌려준다. 패싯도 `FacetAxisResult`로 정규화한다.
- 프로토타입은 자료 C 계약과 **거의 일치**한다(관측된 실제 호출 `POST /search/prior-art/muhayu-patents {searchQuery, fetchOffset, fetchCount, orderByQuery, statuses, docTypes, countries}` 와 동일 형태).

### 1.2 두 명세 간 주요 차이

| 항목 | 자료 A (ISE bigtext) | 자료 C (BFF, 실제) |
|---|---|---|
| 엔드포인트 | `/search/bigtext(-group-by…)` | `/resources/search/prior-art/{muhayu-patents/papers, patent/paper-facets}` |
| 표기 | snake_case, `data{}` 봉투 | camelCase, 봉투 없음 |
| 검색식 | `search_query` (필터 포함해 직접 작성) | `searchQuery` + 구조화 필터 파라미터 분리 |
| 필터(특허) | select/filter query에 포함 | `statuses[] docTypes[] countries[] ipcMain[] cpcMain[] dateFrom/To applicationYears[]` |
| 필터(논문) | 동일하게 query 포함 | `venues[] years[] yearFrom/To` |
| 응답 | `field_name[]`+`result[][]` 원시 | `hits[].patent/paper`(타입드) + `columns{}` 패스스루 |
| 패싯 | group-by/string-list-group-by(gsize/gkey) | `facets.{축}: {items[{key,count}], totalGroups, fetched, error}` |
| 스니펫 하이라이트 | `keyword_char_offset` 제공 | BFF hit엔 미노출(프런트는 클라이언트 하이라이트) |

### 1.3 요구사항(B) ↔ 실제 응답(C) 불일치 — 확인 필요

| 내용 | B(동작방식) | C(실제 BFF) | 판정 |
|---|---|---|---|
| 특허 패싯 축 | 국가·**권리상태**·IPC주분류·CPC주분류·출원연도 (`검색-150`) | `countryCode, ipcMain, cpcMain, applicationYear` | ⚠ **권리상태(register_status) 패싯이 실제 응답에 없음** |
| 논문 국가 | 국가 개념 없음(`검색-211`) | `Paper.countryCode: null` | 일치(항상 null) |

### 1.4 프로토타입 UI에는 있으나 API(A·C)에 정의가 없는 항목 (⚠ 목업 전용)

검색어 입력·결과 목록(In-범위)에서 **웹에 렌더되지만 대응 API 필드가 없는** 요소는 각 페이지 매핑표에서 `⚠ API 미정의(목업)`로 표기한다. 대표적으로:

- **특허 목록**: 원문 PDF URL(원문 버튼), 명칭 키워드 하이라이트(클라이언트 계산).
- **논문 목록**: 권(volume)·호(issue)·시작/끝 페이지·발행 월·논문유형(thesis)·학위수여기관 — `Paper` 타입 미정의.
- **공통**: 라이브러리 저장, 결과 CSV(클라이언트 생성), 검색 히스토리(로컬 상태), 특허↔논문 교차검색 — 모두 프런트 상태/클라이언트 기능(백엔드 API 불필요).
- (상세 전용 ⚠목업 항목 — 도면·패밀리·인용·주소류·구조화 등 — 은 §5 보류로 이관)

### 1.5 프로토타입(D) ↔ 실제 데모(C) 차이 — 확인/보완 필요

| 항목 | 실제 데모(C, search-behavior.md) | 프로토타입(D) | 판정 |
|---|---|---|---|
| 범위(모드) 매핑 | `key`→`KEY:` = 명칭+**초록**+**대표청구항**, `bib_full`→`TAC:` = 명칭+초록+전체청구항, `full`→`DSC:` = **상세설명(전문)** | `KEY`=명칭+요약+독립항 / `TAC`=명칭+요약+전체청구항 / `DSC`=명칭+요약+전체청구항+상세설명 | ⚠ **용어·구성 상이**. 실제: 초록/대표청구항, DSC=상세설명 단독. 프로토타입은 요약/독립항, DSC=전문 합산(사용자 지시 반영분). 라벨·매핑 정합 확인 필요 |
| 검색식 패스스루 | `search_query`는 1바이트도 변형 금지, 메타필터는 `select_query`(SQL WHERE)로 분리 | 동일(자유어 불변 + 필드 AND 누적) | 일치 |
| 상태관리(SSOT) | **URL 쿼리스트링이 단일 진실원**(q·country·state·from·to·mode·paperMode·sort·page·pageSize·selected·facets), 공유링크/뒤로가기/새로고침 복원(`검색-80~82`) | React 상태(해시 탭). URL 직렬화·공유·복원 **미구현** | ⚠ 프로토타입 미구현(데모 핵심 규칙) → §3.5 |
| 재검색 재요청 | URL 무변경 submit 시 명시적 invalidate(`검색-52`) | 동일 개념(항상 재실행) | 일치 |
| 검색필드 유지(E22) | 입력값만 비우고 활성필드 구성 유지, 디폴트 복원은 초기화만 | 동일 | 일치 |
| WIPS ON 식(.IDENT./@필드) | 전문가식으로 보고 모드 미부착·원본 통과 | applyScope가 선두 자유어에만 래핑(필드지정식 불변) | 대체로 일치(WIPS 식별자 특례는 미검증) |

## 2. BFF 계약 요약 (자료 C — 매핑 기준)

**공통**: `POST /resources/search/prior-art/…`, `Content-Type: application/json`, 실패 시 `{message}`.

### 2.1 특허 검색 `/muhayu-patents`
- 요청 `PatentSearchRequest`: `searchQuery`, `fetchOffset?`, `fetchCount?`(기본 20), `orderByQuery?`, `statuses?[]`, `docTypes?[]`, `dateFrom?`, `dateTo?`, `countries?[]`, `ipcMain?[]`, `cpcMain?[]`, `applicationYears?[]`
- 응답 `PatentSearchResponse`: `totalCount`, `fetchCount`, `hits[]`(`PatentHit{ patent, dbfile, rowid }`)
- `Patent`: `docid, inventionTitle, inventionTitleEng, abstract, applicant, inventorName, agentName, applicationNumber, applicationDate, publicationDate, registerDate, openDate, searchBaseDate, ipcMain, cpcMain, registerStatus, countryCode, documentKind, columns{60+}`

### 2.2 논문 검색 `/muhayu-papers`
- 요청 `PaperSearchRequest`: `searchQuery`, `fetchOffset?`, `fetchCount?`, `orderByQuery?`, `yearFrom?`, `yearTo?`, `venues?[]`, `years?[]`
- 응답 `PaperSearchResponse`: `totalCount`, `fetchCount`, `hits[]`(`PaperHit{ paper, dbfile, rowid }`)
- `Paper`: `paperId, title, titleEng, abstract, authors[], journal, publicationYear, publicationDate, keywords[], internalFileUrl, externalFileUrl, citationCount, countryCode(null), score`

### 2.3 패싯 `/patent-facets`, `/paper-facets`
- 요청: 검색요청과 동일 필터(검색어+필터).
- 특허 응답 `facets: { countryCode, ipcMain, cpcMain, applicationYear }` (각 `FacetAxisResult{ items[{key,count}], totalGroups, fetched, error }`)
- 논문 응답 `facets: { journal, publicationYear }`

### 2.4 검색식 표기(자료 B·프로토타입 공통)
- 필드절 `CODE:(값)`, 일자 `CODE:([from ~ to])`, 범위코드 `KEY`(명칭+요약+독립항)/`TAC`(명칭+요약+전체청구항)/`DSC`(명칭+요약+전체청구항+상세설명). 자유어는 선두에만 범위 적용(`검색-21`), 사용자 직접 필드식은 불변(`검색-22`).
- 필드코드 카탈로그는 자료 C `patentSearchField.ts`(TI/AB/CL/CLI/CLA/KEY/TAC/DSC/…/IPCM/CPCM/AP/INV/AG/AN/PN/RN/AD/PD/RD 등 70+), `paperSearchField.ts`.

## 3. 검색 기능명세 (상세 제외 — 입력·목록·공통)

표기: 요소 → **[API]** 소스. `client`=프런트 계산/상태, `⚠목업`=API 미정의. 서술 순서: ① 목적 → ② 화면 구성요소 → ③ 기능·상호작용 → **③′ 기능 설명(사전조건·사후결과)** → **③″ 목록 항목 구성(목록 화면 한정 — 메타 배열 순서·방식·누락 처리)** → ④ 요소→API 매핑 → ⑤ 상태/URL → ⑥ 예외 → ⑦ 접근성.

> **기능 설명 정의**: 각 기능을 **사전조건(그 기능을 쓰기 위해 먼저 충족돼야 하는 상태·입력)** 과 **사후결과(그 기능을 쓴 뒤 발생하는 일·상태 변화)** 로 기술한다. 이는 개발·QA가 "언제 이 기능이 동작 가능하고, 실행하면 무엇이 바뀌는가"를 판단하는 기준이 된다.

### 3.1 특허 · 검색어 입력 (`PatentInput`)

**① 목적**: 변리사가 검색식·필터를 구성해 특허 검색을 실행한다. (요구사항 §2)

**② 화면 구성요소**
- 메타필터: **국가**(전체/KR/US/JP/CN/EP/PCT/DE/GB + 추가국가) · **문헌종류**(전체/공개/등록/실용공개/실용등록) · **기간**(전체/최근5·10·20년 + 일자범위) · **상태**(전체/A:출원·심사중·등록 / I:거절·소멸·취하·포기·무효)
- 검색어 범위 탭(KEY/TAC/DSC) · 입력방식 토글(일반/편집기)
- 검색식 입력창 + 우측 검색/초기화 버튼
- 항목별 상세검색(필드별 조합 · 펼침/접힘 · 필드 추가/확장 모달)
- 검색 히스토리(→ §3.5)

**③ 기능·상호작용**
- 활성 칩은 체크(✓)+색상. "전체"는 라디오형(해제 불가). (`검색-40~42`)
- 검색어 범위: 명칭+요약+독립항 / 명칭+요약+전체청구항 / 명칭+요약+전체청구항+상세설명 (`검색-20`, 라벨 상세는 §1.5 확인 대상)
- 입력방식: 일반모드(textarea) / 편집기모드(구문강조+실시간 진단, 실행 차단 안 함) (`검색-11·12`)
- 항목별 필드에 값 입력 시 검색식에 AND 누적(`검색-60~63`), 자유어는 선두에만 범위코드 래핑(`검색-21`), 사용자 직접 필드식은 불변(`검색-22`). 실행 후 입력값만 비우고 활성필드 구성 유지(`검색-70~72`)

**③′ 기능 설명 — 사전조건 · 사후결과**

| 기능 | 사전조건 | 사후결과 |
|---|---|---|
| 검색 실행 | 특허 검색 탭 진입 · 검색식 입력창 또는 항목별 필드에 값 1개 이상 | `searchQuery` 조립 후 POST → 결과 목록(§3.2)으로 전환 · 검색 히스토리 1건 추가 · (연동 시) URL 쿼리 직렬화 |
| 필터 칩 토글(국가/문헌종류/기간/상태) | 화면 진입 | 선택 상태 반영("전체"=라디오, 해제 불가) · 다음 검색 payload에 `countries/docTypes/statuses/dateFrom·To` 포함 |
| 항목별 필드 값 입력 | 항목별 상세검색 펼침 · 필드 선택됨 | 검색식에 `CODE:(값)` AND 누적 · 실행 후 입력값만 비우고 필드 구성 유지 |
| 검색어 범위 전환(KEY/TAC/DSC) | 화면 진입 | 선두 자유어에 적용될 범위코드 변경(사용자 직접 필드식은 불변) |
| 입력방식 전환(일반/편집기) | 화면 진입 | 표시 방식 전환 · 편집기모드는 실시간 진단 표시(실행은 차단하지 않음) |
| 초기화 | — | 입력값·활성필드를 디폴트로 복원 |

**④ 화면요소 → API 매핑**

| 요소 | 소스 |
|---|---|
| 검색 실행 payload `searchQuery` | 검색식(입력창) + 항목별필드 AND 누적 + 범위코드 래핑 → `PatentSearchRequest.searchQuery` |
| 국가 칩 | → `countries[]` |
| 문헌종류 칩 | → `docTypes[]` |
| 기간(프리셋/일자) | → `dateFrom`/`dateTo` |
| 상태(A/I 칩) | → `statuses[]` |
| 정렬 기본 | → `orderByQuery`(예 `application_date DESC`, 기본 `$SCORE DESC`) |
| 필드코드 목록/타입 | 자료 C `patentSearchField.ts`(text/date-range/ipc) |
| 검색 히스토리·즐겨찾기 | `client` |
| 키워드 추천/파인더 | `⚠목업`(현재 비노출) |

**⑤ 상태/URL**: 검색 실행 시 q·country·state·from·to·mode·필드구성이 URL 쿼리에 직렬화되어야 함(데모 SSOT, 프로토타입 미구현 → §3.5). **⑥ 예외**: 빈 검색식 실행 방지/경고. **⑦ 접근성**: 칩 `aria-pressed`+✓(색상만으로 선택 구분 금지, WCAG 1.4.1).

### 3.2 특허 · 결과 목록 (`PatentResults` / `TableResults`)

**① 목적**: 검색 결과를 표/카드로 보여주고 좁히기·정렬·페이지 이동·선택 저장. (요구사항 §4·§5)

**② 화면 구성요소**
- 상단 요약바: 적용 검색식 칩 · **총 건수** · → 논문으로(교차검색, §3.5)
- 데스크톱 표 / 모바일 카드. 컬럼: 체크박스·상태·문헌번호·원문(PDF)·발명의명칭(+새 탭)·출원인·출원일·IPC (`검색-90`)
- 정렬(출원일/공개일/등록일 + 방향) · perPage · CSV 다운로드 · 페이지네이션
- 패싯 사이드바/필터바: 국가·현재권리상태·IPC·CPC·출원연도

**③ 기능·상호작용**
- 패싯 체크 후 실행(`검색-150~155`), 새 검색 시 1페이지로(`검색-120~122`)
- 체크박스 다중 → 라이브러리 일괄 저장(§3.5) · 로딩/0건/오류 상태(`검색-140~142`)
- 행/카드 클릭 → 오버레이 미리보기 열림(**미리보기 내용 명세는 §5 보류**; 목록 측 동작은 "선택 인덱스 관리 + 패널 오픈"까지가 In-범위)
- 상태값은 전부 평문 표시(색 Badge 미사용) — [상태값 정의서](상세페이지-상태값-정의.md) 정책

**③′ 기능 설명 — 사전조건 · 사후결과**

| 기능 | 사전조건 | 사후결과 |
|---|---|---|
| 목록 렌더 | 검색 실행 완료(응답 수신) | `totalCount`·`hits`를 표/카드로 렌더 |
| 패싯 적용(국가/권리상태/IPC/CPC/출원연도) | 결과 1건 이상 · 패싯 로드됨 | 체크한 축을 필터에 추가해 재검색 → 1페이지로 이동 |
| 정렬/페이지/perPage 변경 | 결과 존재 | `orderByQuery`/`fetchOffset`/`fetchCount` 갱신 후 재요청 |
| 다중 선택 저장 | 체크박스 1개 이상 선택 | 선택 특허를 라이브러리에 저장하고 체크 해제 |
| CSV 내보내기 | 결과 1건 이상 | 현재 `hits` → CSV 생성·다운로드 |
| 행/카드 선택 → 오버레이 | 결과 행 존재 | 선택 인덱스 설정 · 미리보기 패널 오픈(**패널 내용은 §5 보류**) |
| → 논문 교차검색 | 현재 검색어 존재 | 논문 검색으로 검색어 이관·화면 전환 |

**③″ 목록 항목 구성 (메타데이터 배열 순서·방식·누락 처리)**

- **데스크톱(표)** — 고정 컬럼, 셀당 1필드. 순서: ① 체크박스 → ② 상태(평문) → ③ 문헌번호(mono) → ④ 원문 PDF(아이콘) → ⑤ 발명의 명칭(+새 탭 열기) → ⑥ 출원인 → ⑦ 출원일 → ⑧ IPC
- **모바일(카드)** — ① 체크박스 → ② `상태 · 문헌번호`(한 줄) → ③ 제목(2줄 클램프) → ④ `출원인 · 출원일 · IPC`(한 줄, ` · ` 연결) → ⑤ 원문 PDF·새 탭 버튼
- **문헌번호 구성**: `국가코드 + 번호 + 문헌종류`, 번호는 우선순위(등록 > 공고 > 공개 > 출원)로 1개 확정(자료 B)
- **누락 예외처리**: 표 셀은 값이 없으면 **빈 셀**로 렌더(현재 폴백 텍스트 없음). 모바일 메타라인은 ` · ` **고정 연결**이라 값 누락 시 구분자만 남음 → **⚠ 연동 시 누락 필드 생략/'—' 폴백 규칙 정의 필요**
- CSV 컬럼: 문헌번호·명칭·출원인·출원일·상태·IPC

**④ 화면요소 → API 매핑**

| 요소 | 소스 |
|---|---|
| 총 건수 | `PatentSearchResponse.totalCount` |
| 행 목록 | `hits[].patent` |
| 상태(평문) | `patent.registerStatus` |
| 문헌번호 | `patent.countryCode`+공백+번호(우선순위 등록>공고>공개>출원: `registerNumber`>`publicationNumber`>`openNumber`>`applicationNumber`, `columns`에서)+공백+`documentKind` (자료 B 규칙) |
| 발명의 명칭 | `patent.inventionTitle`(+`inventionTitleEng`) |
| 출원인 | `patent.applicant` |
| 출원일 | `patent.applicationDate` |
| IPC | `patent.ipcMain` |
| 원문 PDF 아이콘 | `⚠목업`(원문 URL 필드 미정의; `columns` 내 원문키 확인 필요) |
| 명칭 키워드 하이라이트 | `client`(요구 `검색-100`; BFF엔 offset 미노출) |
| 정렬/페이지/perPage | → `orderByQuery`/`fetchOffset`/`fetchCount` |
| 필터칩·패싯 건수 | `/patent-facets` `facets.{countryCode,ipcMain,cpcMain,applicationYear}` (⚠ 권리상태 패싯은 응답에 없음 → 상태는 검색요청 `statuses`로만) |
| CSV | `client`(hits→CSV) |
| 체크 저장 | `client`(라이브러리, §3.5) |
| 행 선택→오버레이 | `client`(선택 인덱스; 패널 내용은 §5) |

**⑤ 상태/URL**: sort·page·pageSize·facets·selected(선택 문서)가 URL에 반영되어야 함(§3.5). **⑥ 예외**: 로딩 스켈레톤 / 0건 안내 / 검색 오류 메시지(`검색-140~142`). **⑦ 접근성**: 표 헤더 스코프, 패싯 체크박스 라벨.

### 3.3 논문 · 검색어 입력 (`PaperInput`)

**① 목적**: 논문 검색식·필터 구성·실행. 흐름은 특허와 대칭(`검색-213`).

**② 화면 구성요소 / ③ 기능**
- 발행연도 필터(전체/최근1·5·10년 + 연도 from~to) (`검색-211`, 국가 없음)
- 검색어 범위 탭: 제목+초록 / 제목+초록+전문
- 검색식 입력창 + 검색/초기화
- 항목별 상세검색(제목·초록·키워드·전문·저자 등, 펼침/접힘) → AND 누적
- 검색 히스토리(특허와 동일 · §3.5)

**③′ 기능 설명 — 사전조건 · 사후결과**

| 기능 | 사전조건 | 사후결과 |
|---|---|---|
| 검색 실행 | 논문 검색 탭 진입 · 검색식 또는 항목별 필드에 값 1개 이상 | `PaperSearchRequest` 조립·POST → 논문 목록(§3.4) 전환 · 검색 히스토리 추가 |
| 발행연도 필터 | 화면 진입 | `yearFrom`/`yearTo`(또는 `years[]`) 반영(국가 필터 없음) |
| 항목별 필드 값 입력 | 상세검색 펼침 · 필드 선택됨 | 검색식 AND 누적 · 실행 후 입력값만 비움 |
| 검색어 범위 전환(제목+초록 / +전문) | 화면 진입 | 선두 자유어 적용 범위 변경 |
| 초기화 | — | 입력값·필드 구성 복원 |

**④ 화면요소 → API 매핑**

| 요소 | 소스 |
|---|---|
| 검색 payload | → `PaperSearchRequest.searchQuery` (범위코드 래핑 + 필드 AND 누적) |
| 발행연도(프리셋/from~to) | → `yearFrom`/`yearTo` (또는 `years[]`) |
| 정렬 | → `orderByQuery` |
| 필드코드 | 자료 C `paperSearchField.ts`(제목·초록·키워드·전문·저자) |

**⑤ 상태/URL**: q·paperMode·yearFrom·yearTo가 URL에(§3.5). **⑦ 접근성**: 칩 `aria-pressed`+✓.

### 3.4 논문 · 결과 목록 (`PaperResults`)

**① 목적**: 논문 결과 카드 목록·좁히기·정렬·페이지·선택 저장.

**② 화면 구성요소 / ③ 기능**
- 요약바: 검색식 칩·총 건수·→ 특허로(교차검색, §3.5)
- 카드: 제목(+대등제목) · 메타라인(저자, 저널 권(호), 페이지, 발행연도(월)) · **초록**(앞에 "초록" 라벨) · 새 탭 열기 · 체크박스
- 필터바/패싯: 저널·발행연도
- 정렬·perPage·CSV(제목·저자·저널·발행연도) · 페이지네이션 · 다중선택 저장
- 카드 클릭 → 오버레이 미리보기(**내용 명세는 §5 보류**; 목록 측은 선택/패널 오픈까지)

**③′ 기능 설명 — 사전조건 · 사후결과**

| 기능 | 사전조건 | 사후결과 |
|---|---|---|
| 목록 렌더 | 검색 실행 완료(응답 수신) | `totalCount`·`paper` 카드 렌더 |
| 패싯 적용(저널/발행연도) | 결과 1건 이상 · 패싯 로드됨 | 저널/연도 필터 추가해 재검색 → 1페이지로 |
| 정렬/페이지/perPage 변경 | 결과 존재 | `orderByQuery`/`fetchOffset` 갱신 후 재요청 |
| 다중 선택 저장 | 체크 1개 이상 | 선택 논문을 라이브러리에 저장하고 체크 해제 |
| CSV 내보내기 | 결과 1건 이상 | 카드(제목·저자·저널·발행연도) → CSV 다운로드 |
| 카드 선택 → 오버레이 | 카드 존재 | 선택 인덱스 설정·패널 오픈(**내용은 §5 보류**) |
| → 특허 교차검색 | 현재 검색어 존재 | 특허 검색으로 검색어 이관·전환 |

**③″ 목록 항목 구성 (메타데이터 배열 순서·방식·누락 처리)**

- **카드 순서**: ① 체크박스 → ② 제목(하이라이트) + 새 탭 열기 → ③ 메타라인 → ④ 초록(**있을 때만** · 앞에 "초록" 라벨 + 2줄 클램프)
- **메타라인 배열(`paperMetaLine`)**: `저자, [저널명 | 학위수여기관] 권(호), 시작-끝페이지, 발행연도(월)` — **콤마(`, `)로 연결하고 없는 항목은 생략**
  - 출처: `paperType==='thesis'` → 학위수여기관, 그 외 → 저널명
  - 권/호: `volume` 있으면 ` {권}`, `issue` 있으면 `({호})` 부가
  - 페이지: 시작·끝 모두 있으면 `시작-끝`, 시작만 있으면 `시작`
  - 발행: `month` 있으면 `연도. 월`, 없으면 `연도`
- **누락 예외처리**: 각 요소 부재 시 **콤마 목록에서 생략**(빈 자리 없음). 초록 부재 시 **초록 줄 자체를 렌더하지 않음**. (오버레이/상세 패널에서는 부재 시 `-` 표기)
- CSV 컬럼: 제목·저자·저널·발행연도

**④ 화면요소 → API 매핑**

| 요소 | 소스 |
|---|---|
| 총 건수 | `PaperSearchResponse.totalCount` |
| 카드 목록 | `hits[].paper` |
| 제목/대등제목 | `paper.title`/`paper.titleEng` |
| 저자 | `paper.authors[]` |
| 저널명 | `paper.journal` |
| 발행연도 | `paper.publicationYear` (발행일 `publicationDate`) |
| 권(호)·시작/끝페이지·발행 월·학위수여기관 | `⚠목업`(Paper 타입 미정의) |
| 초록(+"초록" 라벨) | `paper.abstract` |
| 초록 하이라이트 | `client` |
| 필터/패싯 건수 | `/paper-facets` `facets.{journal, publicationYear}`; 요청 `venues[]`/`years[]` |
| CSV/체크 저장 | `client` |
| 카드 선택→오버레이 | `client`(선택 인덱스; 패널 내용은 §5) |

**⑤ 상태/URL**: sort·page·pageSize·facets·selected(§3.5). **⑥ 예외**: 로딩/0건/오류. **⑦ 접근성**: 카드 포커스·초록 라벨.

### 3.5 공통 · 횡단 기능

검색 도메인 전체에 걸치는 기능. 대부분 프런트 상태(`client`)이나, **URL SSOT는 데모의 핵심 규칙이며 연동 시 반드시 정의**해야 한다.

| 기능 | 사전조건 | 사후결과 | 소스 / 비고 |
|---|---|---|---|
| **검색 히스토리** | 검색 1회 이상 실행됨(기록 존재) | 항목 선택 시 해당 검색식으로 재실행 · ★즐겨찾기 토글 저장(최대 100건) | `client`(로컬 store) |

| **재검색 재요청** | 동일 검색 조건으로 재실행 | URL 무변경이어도 명시적 invalidate·재요청(`검색-52`) | `client` |

| **특허 ↔ 논문 교차검색** | 현재 검색어 존재 | 반대 도메인 검색으로 검색어 이관·화면 전환 | `client`(별도 API 없음) |
| **CSV 내보내기** | 결과 1건 이상 | 현재 hits → CSV 생성·다운로드 | `client` |
| **로딩/0건/오류 상태** | 검색 실행됨 | 응답 대기=로딩 · 0건=안내 · 실패=오류 메시지(`검색-140~142`) | `client` + 응답 성공/실패 |

## 4. 종합 — 검색 기능 연동 시 정의가 필요한 항목 (상세 제외)

In-범위(입력·목록·공통)에서 실제 API 연동 시 확정해야 할 항목:

1. **특허 원문 PDF URL** — 목록 원문 버튼(`⚠목업`). `columns` 내 원문키 존재 여부 확인.
2. **특허 권리상태 패싯** — 요구사항(`검색-150`)엔 있으나 `/patent-facets` 응답에 없음(추가 or 상태는 검색필터 `statuses`로만).
3. **논문 서지 보강**(권/호/페이지/월/논문유형/학위수여기관) — 목록 메타라인에 렌더되나 `Paper` 타입 미정의 → 타입 확장 필요.
4. **URL 상태(SSOT) 규칙** — 데모 핵심 규칙(§3.5)이나 프로토타입 미구현. 라우팅·직렬화·복원 정의 필요.
5. **검색어 범위(모드) 라벨·매핑 정합**(§1.5) — 데모(초록/대표청구항, DSC=상세설명)와 프로토타입(요약/독립항, DSC=전문합산) 용어 차이 확정.
6. **명칭·초록 하이라이트** — 클라이언트 계산(BFF에 offset 미노출) — 유지/서버제공 여부 결정.

## 5. 보류 — 상세페이지 (서비스 DB 설계 후)

> **보류 사유**: 특허 **수집 DB·서비스 DB가 설계 전**이다. 상세/오버레이에 노출할 필드·구조는 수집 스키마가 확정돼야 정의할 수 있으므로, 상세 명세는 DB 설계 완료 후 착수한다.

**보류 대상 (4)**: 특허 오버레이 미리보기 · 특허 전체 상세페이지 · 논문 오버레이 미리보기 · 논문 전체 상세페이지.
- 이들은 검색 hit(`patent`/`paper`+`columns`)를 재사용해 렌더하며 별도 상세 API가 없다. 목록 측의 "선택→패널 오픈" 동작은 §3.2·§3.4에 In-범위로 정의돼 있고, **패널·페이지의 내용(필드/섹션) 명세만 보류**한다.

**DB 설계 시 정의해야 할 상세 필드·섹션 (후속 작업 대상)**
- 특허: 도면(이미지·대표도면·부호의 설명) · 패밀리 정보 · 인용/피인용 · 서지 보강(공개/공고/등록 번호, 우선권주장일·심사청구일·존속만료일·권리변동) · 출원인/발명자/대리인 주소·식별코드 · 상세설명/청구항 구조화(소제목·독립/종속) · 국가고유(FI/FTERM/TEMA·UPC·EPC·지정국·관련출원 등)
- 논문: 영문 저널/영문 초록 · 분야 · 학위수여기관 · 인용 포맷(APA/MLA/Chicago/Harvard/BibTeX, 클라이언트 생성) · 관련논문
