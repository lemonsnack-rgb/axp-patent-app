# AXP 선행기술검색 프로토타입 — 기능정의서 (검색 기능 · 상세 제외)

문서 목적: 이번에 만든 **목업/프로토타입(웹에서 실제 렌더되는 기능 기준)**의 기능을, **UI 상 이용자의 이용 흐름**과 **표출되는 정보의 표시 기준** 중심으로 화면별 정의한다. 각 화면 요소의 데이터 출처(API)는 **연동 시 참고**로 병기하되, 목업 단계이므로 확정 구현 스펙으로 보지 않는다.

> ⚠️ **범위(2026-07-06 재조정)**: **특허 수집·서비스 DB가 아직 설계 전**이다. 상세페이지는 수집 필드가 확정돼야 정의할 수 있으므로, 본 문서는 **상세 비의존 기능(검색어 입력·결과 목록·공통)**을 먼저 정의한다. **오버레이 미리보기와 전체 상세페이지는 보류(상세페이지)**로 둔다. (상세 필드 후보는 `상세페이지-수집필드-대조.md`·`상세페이지-상태값-정의.md` 문서에 별도 정리)

원칙(작성 규칙):
- **UI에 실제 렌더되는 것만 기재한다(내용 창작 금지).** 코드로 확인되지 않는 동작·필드는 추정해 쓰지 않는다. 미구현·미확정은 그대로 `⚠ 미구현 / 연동 시 정의`로 표기한다.
- **API 명세에만 있고 UI에 없는 항목의 누락은 허용**한다(문서에서 생략 가능).
- **UI(웹 렌더)에 있는데 이 문서에서 누락되면 안 된다.** → 대상 UI 요소는 전부 기술하고, 대응 API가 없으면 `⚠ API 미정의(목업)`로 명시한다.

> **구성**: 아래 세 부분은 **각각 독립적으로 게시**될 수 있다. ① 배경 및 참고자료(맥락·근거) · ② 메뉴·화면별 기능명세(구현 대상) · ③ 부록(연동 필요 항목·보류).

> 핸드오프 컨텍스트. 이 부분은 요구사항·API 구조를 이해하기 위한 **참고 자료**이며, 구현 대상 명세는 「메뉴·화면별 기능명세」에 있다.

## 명세 비교 및 차이 (분석)

### API는 2계층 구조다

```
[UI/프로토타입]
    │  호출
    ▼
(BFF) /resources/search/prior-art/{muhayu-patents | muhayu-papers | patent-facets | paper-facets}   ← prior-art-search BFF (실제 UI 계약)
    │  래핑
    ▼
(ISE) /search/bigtext , /search/bigtext-group-by , /search/bigtext-string-list-group-by            ← bigtext 매뉴얼 (저수준 API)
```

- **bigtext 매뉴얼(저수준)** 은 최하위 ISE 검색엔진 wrapper를 문서화한 것이다. 필터(국가·상태·기간)를 별도 파라미터로 받지 않고 `search_query`/`select_query`/`filter_query`에 접어 넣는다. 응답도 `field_name[]`+`result[][]` 원시 배열이다.
- **prior-art-search BFF** 는 UI 친화 계약이다. 필터를 **구조화 파라미터**(statuses/docTypes/countries/ipcMain/cpcMain/dates/years/venues)로 받고, 응답을 **타입드 객체**(`Patent`/`Paper`) + `columns`(60+ 컬럼 패스스루)로 돌려준다. 패싯도 `FacetAxisResult`로 정규화한다.
- 프로토타입은 BFF 계약과 **거의 일치**한다(관측된 실제 호출 `POST /search/prior-art/muhayu-patents {searchQuery, fetchOffset, fetchCount, orderByQuery, statuses, docTypes, countries}` 와 동일 형태).

### 두 계층 간 주요 차이

| 항목 | bigtext 매뉴얼 (ISE 저수준) | prior-art-search BFF (실제) |
|---|---|---|
| 엔드포인트 | `/search/bigtext(-group-by…)` | `/resources/search/prior-art/{muhayu-patents/papers, patent/paper-facets}` |
| 표기 | snake_case, `data{}` 봉투 | camelCase, 봉투 없음 |
| 검색식 | `search_query` (필터 포함해 직접 작성) | `searchQuery` + 구조화 필터 파라미터 분리 |
| 필터(특허) | select/filter query에 포함 | `statuses[] docTypes[] countries[] ipcMain[] cpcMain[] dateFrom/To applicationYears[]` |
| 필터(논문) | 동일하게 query 포함 | `venues[] years[] yearFrom/To` |
| 응답 | `field_name[]`+`result[][]` 원시 | `hits[].patent/paper`(타입드) + `columns{}` 패스스루 |
| 패싯 | group-by/string-list-group-by(gsize/gkey) | `facets.{축}: {items[{key,count}], totalGroups, fetched, error}` |
| 스니펫 하이라이트 | `keyword_char_offset` 제공 | BFF hit엔 미노출(프런트는 클라이언트 하이라이트) |

### 요구사항(동작방식 개발노트) ↔ 실제 응답(BFF) 불일치 — 확인 필요

| 내용 | 동작방식 개발노트 | prior-art-search BFF | 판정 |
|---|---|---|---|
| 특허 패싯 축 | 국가·**권리상태**·IPC주분류·CPC주분류·출원연도 | `countryCode, ipcMain, cpcMain, applicationYear` | ⚠ **권리상태(register_status) 패싯이 실제 응답에 없음** |
| 논문 국가 | 국가 개념 없음 | `Paper.countryCode: null` | 일치(항상 null) |

### 프로토타입 UI에는 있으나 API(bigtext 매뉴얼·BFF)에 정의가 없는 항목 (⚠ 목업 전용)

검색어 입력·결과 목록에서 **웹에 렌더되지만 대응 API 필드가 없는** 요소는 각 매핑표에서 `⚠ API 미정의(목업)`로 표기한다. 대표적으로:

- **특허 목록**: 원문 PDF URL(원문 버튼), 명칭 키워드 하이라이트(클라이언트 계산).
- **논문 목록**: 권(volume)·호(issue)·시작/끝 페이지·발행 월·논문유형(thesis)·학위수여기관 — `Paper` 타입 미정의.
- **공통**: 라이브러리 저장, 결과 CSV(클라이언트 생성), 검색 히스토리(로컬 상태), 특허↔논문 교차검색 — 모두 프런트 상태/클라이언트 기능(백엔드 API 불필요).
- (상세 전용 ⚠목업 항목 — 도면·패밀리·인용·주소류·구조화 등 — 은 「부록」의 보류 항목으로 이관)

## BFF 계약 요약 (prior-art-search BFF — 매핑 기준)

**공통**: `POST /resources/search/prior-art/…`, `Content-Type: application/json`, 실패 시 `{message}`.

### 특허 검색 `/muhayu-patents`
- 요청 `PatentSearchRequest`: `searchQuery`, `fetchOffset?`, `fetchCount?`(기본 20), `orderByQuery?`, `statuses?[]`, `docTypes?[]`, `dateFrom?`, `dateTo?`, `countries?[]`, `ipcMain?[]`, `cpcMain?[]`, `applicationYears?[]`
- 응답 `PatentSearchResponse`: `totalCount`, `fetchCount`, `hits[]`(`PatentHit{ patent, dbfile, rowid }`)
- `Patent`: `docid, inventionTitle, inventionTitleEng, abstract, applicant, inventorName, agentName, applicationNumber, applicationDate, publicationDate, registerDate, openDate, searchBaseDate, ipcMain, cpcMain, registerStatus, countryCode, documentKind, columns{60+}`

### 논문 검색 `/muhayu-papers`
- 요청 `PaperSearchRequest`: `searchQuery`, `fetchOffset?`, `fetchCount?`, `orderByQuery?`, `yearFrom?`, `yearTo?`, `venues?[]`, `years?[]`
- 응답 `PaperSearchResponse`: `totalCount`, `fetchCount`, `hits[]`(`PaperHit{ paper, dbfile, rowid }`)
- `Paper`: `paperId, title, titleEng, abstract, authors[], journal, publicationYear, publicationDate, keywords[], internalFileUrl, externalFileUrl, citationCount, countryCode(null), score`

### 패싯 `/patent-facets`, `/paper-facets`
- 요청: 검색요청과 동일 필터(검색어+필터).
- 특허 응답 `facets: { countryCode, ipcMain, cpcMain, applicationYear }` (각 `FacetAxisResult{ items[{key,count}], totalGroups, fetched, error }`)
- 논문 응답 `facets: { journal, publicationYear }`

### 검색식 표기 (동작방식 개발노트·프로토타입 공통)
- 필드절 `CODE:(값)`, 일자 `CODE:([from ~ to])`, 범위코드 `KEY`(명칭+요약+독립항)/`TAC`(명칭+요약+전체청구항)/`DSC`(명칭+요약+전체청구항+상세설명). 자유어는 선두에만 범위 적용, 사용자 직접 필드식은 불변.
- 필드코드 카탈로그는 prior-art-search `patentSearchField.ts`(TI/AB/CL/CLI/CLA/KEY/TAC/DSC/…/IPCM/CPCM/AP/INV/AG/AN/PN/RN/AD/PD/RD 등 70+), `paperSearchField.ts`.

# 메뉴·화면별 기능명세 (AXP 선행기술검색 · 검색 기능)

> **목업 단계 서술 기준**: 지금은 목업/프로토타입 단계이므로, **UI 상 이용자의 이용 흐름**과 **표출되는 정보의 표시 기준**을 중심으로 기술한다. 구체적 구현 방안(엔드포인트·필드 배선 등)은 **정보 출처(연동 시 참고)**에 후보로만 남기고, 확정 스펙으로 보지 않는다.
>
> 각 화면 서술 순서: ① 목적 → **② 구성요소별 정의(화면 위계 순 통합표 — 구성요소·표시 정보·기능·사전조건·사후결과를 한 행에)** → (목록 화면) 목록 항목 구성 → ④ 정보 출처(연동 시 참고) → ⑤ 상태/URL → ⑥ 예외 → ⑦ 접근성.
>
> **② 통합표**는 화면에 나타나는 순서(정보 위계)대로 구성요소를 나열하고, 각 구성요소의 표시 내용·상호작용·사전조건·사후결과를 한 행에 모아 이용 흐름을 위→아래로 읽을 수 있게 한다.
>
> 표기: `client`=프런트 계산/상태, `⚠목업`=대응 API 미정의(연동 시 정의).
>
> **기능 ID(채번)**: 각 구성요소 행에 `도메인-화면-순번` 형식 ID를 부여한다 — 도메인(특허=`PAT` / 논문=`PAP` / 공통=`COM`) · 화면(검색어 입력=`INP` / 결과 목록=`LST`) · 순번(10단위). 이 ID로 **목업 화면 요소와 명세 항목을 연결(부착)**할 수 있다(예: 요소에 `data-spec="PAT-INP-010"`).

> **기능 설명 정의**: 각 기능을 **사전조건(그 기능을 쓰기 위해 먼저 충족돼야 하는 상태·입력)** 과 **사후결과(그 기능을 쓴 뒤 발생하는 일·상태 변화)** 로 기술한다. 개발·QA가 "언제 이 기능이 동작 가능하고, 실행하면 무엇이 바뀌는가"를 판단하는 기준이다.

## 특허 검색

> **이용 흐름**: 검색어·필터 구성 → 검색 실행 → 결과 목록 확인 → (좁히기: 패싯/정렬) → 행 선택 시 미리보기(보류) · 다중 선택 저장 · 논문으로 교차검색.

**③″ 목록 항목 구성 (메타데이터 배열 순서·방식·누락 처리)**

- **카드 순서**: ① 체크박스 → ② 제목(하이라이트) + 새 탭 열기 → ③ 메타라인 → ④ 초록(**있을 때만** · 앞에 "초록" 라벨 + 2줄 클램프)
- **메타라인 배열(`paperMetaLine`)**: `저자, [저널명 | 학위수여기관] 권(호), 시작-끝페이지, 발행연도(월)` — **콤마(`, `)로 연결하고 없는 항목은 생략**
  - 출처: `paperType==='thesis'` → 학위수여기관, 그 외 → 저널명
  - 권/호: `volume` 있으면 ` {권}`, `issue` 있으면 `({호})` 부가
  - 페이지: 시작·끝 모두 있으면 `시작-끝`, 시작만 있으면 `시작`
  - 발행: `month` 있으면 `연도. 월`, 없으면 `연도`
- **누락 예외처리**: 각 요소 부재 시 **콤마 목록에서 생략**(빈 자리 없음). 초록 부재 시 **초록 줄 자체를 렌더하지 않음**. (오버레이/상세 패널에서는 부재 시 `-` 표기)
- CSV 컬럼: 제목·저자·저널·발행연도

**④ 정보 출처 (연동 시 참고 — 목업 단계라 확정 스펙 아님)**

| 요소 | 소스 |
|---|---|
| 총 건수 | `PaperSearchResponse.totalCount` |
| 카드 목록 | `hits[].paper` |

| 저자 | `paper.authors[]` |

| 발행연도 | `paper.publicationYear` (발행일 `publicationDate`) |

## 공통 · 횡단 기능

검색 도메인 전체에 걸치는 기능. 대부분 프런트 상태(`client`)이나, **URL SSOT는 데모의 핵심 규칙이며 연동 시 반드시 정의**해야 한다.

| ID | 기능 | 사전조건 | 사후결과 | 소스 / 비고 |
|---|---|---|---|---|
| `COM-010` | **검색 히스토리** | 검색 1회 이상 실행됨(기록 존재) | 항목 선택 시 해당 검색식으로 재실행 · ★즐겨찾기 토글 저장(최대 100건) | `client`(로컬 store) |
| `COM-020` | **URL 상태(SSOT)** | 검색 실행 or 쿼리 포함 URL 진입 | 상태를 URL(`q·country·state·from·to·mode·paperMode·sort·page·pageSize·selected·facets`)에 직렬화 → 공유·뒤로가기·새로고침 복원 | ⚠ **프로토타입 미구현** — 연동 시 라우팅/직렬화 규칙 정의 필요 |
| `COM-030` | **재검색 재요청** | 동일 검색 조건으로 재실행 | URL 무변경이어도 명시적 invalidate·재요청 | `client` |
| `COM-040` | **라이브러리 저장** | 저장 대상(목록 체크 1개↑) 선택 | 선택 항목을 로컬 라이브러리에 저장하고 체크 해제 | `client`(오버레이·상세 저장 버튼은 보류) |
| `COM-050` | **특허 ↔ 논문 교차검색** | 현재 검색어 존재 | 반대 도메인 검색으로 검색어 이관·화면 전환 | `client`(별도 API 없음) |
| `COM-060` | **CSV 내보내기** | 결과 1건 이상 | 현재 hits → CSV 생성·다운로드 | `client` |
| `COM-070` | **로딩/0건/오류 상태** | 검색 실행됨 | 응답 대기=로딩 · 0건=안내 · 실패=오류 메시지 | `client` + 응답 성공/실패 |

# 부록 (AXP 선행기술검색 · 검색 기능)

## 종합 — 검색 기능 연동 시 정의가 필요한 항목 (상세 제외)

검색어 입력·목록·공통에서 실제 API 연동 시 확정해야 할 항목:

1. **특허 원문 PDF URL** — 목록 원문 버튼(`⚠목업`). `columns` 내 원문키 존재 여부 확인.
2. **특허 권리상태 패싯** — 요구사항엔 있으나 `/patent-facets` 응답에 없음(추가 or 상태는 검색필터 `statuses`로만).
3. **논문 서지 보강**(권/호/페이지/월/논문유형/학위수여기관) — 목록 메타라인에 렌더되나 `Paper` 타입 미정의 → 타입 확장 필요.
4. **URL 상태(SSOT) 규칙** — 데모 핵심 규칙(공통·횡단)이나 프로토타입 미구현. 라우팅·직렬화·복원 정의 필요.
5. **검색어 범위(모드) 라벨·매핑 정합** — 데모(초록/대표청구항, DSC=상세설명)와 프로토타입(요약/독립항, DSC=전문합산) 용어 차이 확정.
6. **명칭·초록 하이라이트** — 클라이언트 계산(BFF에 offset 미노출) — 유지/서버제공 여부 결정.

## 보류 — 상세페이지 (서비스 DB 설계 후)

> **보류 사유**: 특허 **수집 DB·서비스 DB가 설계 전**이다. 상세/오버레이에 노출할 필드·구조는 수집 스키마가 확정돼야 정의할 수 있으므로, 상세 명세는 DB 설계 완료 후 착수한다.

**보류 대상 (4)**: 특허 오버레이 미리보기 · 특허 전체 상세페이지 · 논문 오버레이 미리보기 · 논문 전체 상세페이지.
- 이들은 검색 hit(`patent`/`paper`+`columns`)를 재사용해 렌더하며 별도 상세 API가 없다. 목록 측의 "선택→패널 오픈" 동작은 각 결과 목록 화면에 정의돼 있고, **패널·페이지의 내용(필드/섹션) 명세만 보류**한다.

**DB 설계 시 정의해야 할 상세 필드·섹션 (후속 작업 대상)**
- 특허: 도면(이미지·대표도면·부호의 설명) · 패밀리 정보 · 인용/피인용 · 서지 보강(공개/공고/등록 번호, 우선권주장일·심사청구일·존속만료일·권리변동) · 출원인/발명자/대리인 주소·식별코드 · 상세설명/청구항 구조화(소제목·독립/종속) · 국가고유(FI/FTERM/TEMA·UPC·EPC·지정국·관련출원 등)
- 논문: 영문 저널/영문 초록 · 분야 · 학위수여기관 · 인용 포맷(APA/MLA/Chicago/Harvard/BibTeX, 클라이언트 생성) · 관련논문

**연계 문서 (상세 필드 사전 정리 — DB 설계 입력물)**
- `상세페이지-수집필드-대조.md` — 수집 DB ↔ 상세 필드 대조 + 역방향(UI→명세) 검증
- `상세페이지-상태값-정의.md` — 상세 상태값 표시 정의(현재 완전 평문 정책)
- `목업-조건표.md` — 상세 조건부 목업 필드의 채움 조건(재현 규칙)

> 상세 명세는 위 3개 문서를 입력물로, **수집 DB 스키마 확정 시** 「메뉴·화면별 기능명세」와 동일 템플릿(①~⑦ + 요소→소스 매핑)으로 작성한다.
