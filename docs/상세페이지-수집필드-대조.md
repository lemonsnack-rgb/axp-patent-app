# 특허 상세페이지 ↔ 수집 DB 필드 대조 및 보완 계획 (KR + 공통 우선)

목적: 수집 DB([BLK] 특허 수집 데이터베이스 명세서)가 보유한 필드를 현재 특허 상세페이지 구현과 대조해 **누락/부분 노출** 필드를 도출하고, **이후 Claude Code로 채울 수 있게 구현 가능한 보완 계획**을 제시한다.

대원칙: **수집 데이터 항목은 대부분 서비스(상세)에 출력한다.** 기본값 노출, 제외는 근거 명시.

> ⚠️ 성격: 본 문서와 프로토타입의 `노출-목업` 필드는 **완벽한 구현이 아니라, 개발자에게 "어떤 필드를 어떻게 보여줄지"를 설명하기 위한 전달물**이다. 목업 값의 채움 조건은 [목업-조건표.md](목업-조건표.md) 참조. 실제 서비스에서는 수집 DB/검색 API 값으로 채운다.

## 0. 대조 대상·판정 기준

- A(수집): `검색 API명세서/[BLK] 특허 수집 데이터베이스 명세서` — 공통 정규화 테이블 + KR 고유(우선). JP/US/EP 고유는 §5에 요약.
- B(현재 구현): `src/components/PatentDetail.tsx`(렌더) + `src/types.ts` `PatentResult`(모델) + `src/data/patentSeed.ts`(목업 시드).
- 노출 라벨: `노출-실` / `노출-목업` / **`누락`** / `부분`(카운트/요약만) / `제외`(내부키·수집메타)
- 소스판정(연동 시): `화면`(상세페이지 요소 추가) · `API`(검색 hit `columns` 또는 상세조회 노출) · `둘다`
- 필요성 등급: `P1`(필수·핵심), `P2`(권장), `P3`(선택/특수)

> 데이터 소스 주의: 상세페이지는 검색 hit(`patent`+`columns` 패스스루)로 렌더된다. 수집 DB에 있어도 검색/상세 API가 노출하는지는 별개이므로, 보완 계획에 API 노출 필요 여부를 병기한다.

## 1. 현재 상세페이지 렌더 인벤토리 (B)

`PatentDetail.tsx` 섹션 = 서지사항 · 인명정보 · 요약 · 상세설명 · 청구범위 · 패밀리정보 · 인용·피인용 · 분류코드 · 기타정보 · (우) 도면.
`PatentResult`(목업 시드로 채움): number·country·status·title·applicant·inventors·applicationNo/Date·publicationNo/Date·registerNo/Date·expirationDate·ipc·cpc·rightStatus·rightChange·grade·trial·dispute·rejectionCount·applicantStandard·standardOrg·abstract·repClaim·aiPurpose/Solution/Effect·family(수)·citing/cited(수)·figures[]·refSigns[]·claims[]·citingList/citedList[]·applicantAddress·inventorAddress·applicantCode·priorityDate·examRequestDate·terminationDate·description·agent·agentAddress.
→ 대부분 **목업 데이터**로 렌더(실 API 미연동).

## 3. 누락·보강 필드 요약 (등급순)

- **P1**: 실 도면 이미지 연동(drawing_url/primary_drawing_url/patent_image), 패밀리 개별 문헌 목록, 실 청구항·상세설명(specification 전문/PDF) 연동, 실 인용/피인용 연동. (골격은 화면에 있으나 **목업 → 실데이터** 전환이 핵심)
- **P2**: IPC/CPC 전체 리스트, 우선권 목록, 심사관, 대표출원인·특허고객번호, 선행기술문헌·비특허참고문헌 분리, 권리변동 이력·행정처리(수발신), R&D, 표준특허 보강, 원출원/국제출원, 청구항 수, final_disposal, has_trial/has_ownership_change 정합.
- **P3**: 출원구분·번역문제출일·기술이전희망·지정국·권리이전 이력·실시권등록일·도면수/대표도면 메타·서열목록·KPA_IPC 상세.

## 4. 보완 계획 (바이브코딩 실행 단위)

각 항목은 「① 타입(`PatentResult`/`columns`) ② 목업 시드(`patentSeed.ts`) ③ 화면(`PatentDetail.tsx` 섹션/위치)」 3파트로 구현.

- **BC-1 (P1) 실 도면 연동 준비**: `PatentResult.figures`에 `imageUrl`(대표=primary_drawing_url, 그외=drawing_url/patent_image) 필드 추가 → `DrawingsPanel`이 URL 있으면 `<img>`, 없으면 현행 SVG 폴백. 시드에 sample URL 채움.
- **BC-2 (P1) 패밀리 문헌 목록**: `PatentResult.familyList?: {country, docNumber, date, title}[]` 추가 → 패밀리정보 섹션을 건수 pill + **문헌 목록 표**로. 시드 3~5건.
- **BC-3 (P2) 우선권 목록**: `priorityList?: {country, number, date}[]` → 서지 하단 "우선권" 표. (기존 priorityDate 대체/병기)
- **BC-4 (P2) IPC/CPC 전체**: `ipcList?: string[]`, `cpcList?: string[]` → 분류코드 섹션을 리스트로.
- **BC-5 (P2) 인명 보강**: 심사관 `examiner?`, 대표출원인 `repApplicant?`, 특허고객번호 `customerNo?` → 인명정보 행 추가.
- **BC-6 (P2) 선행기술/비특허 인용 분리**: `citingList` 항목에 `docType`(patent/npl/prior) 세분 → 인용·피인용 섹션에 "선행기술문헌"·"비특허참고문헌" 구분 표기(이미 kind 존재, prior 추가).
- **BC-7 (P2) 권리변동·행정처리 이력**: `rightChangeList?: {type,name,code,date}[]`, `adminProcess?: {docName,date,status,sendRecv}[]` → 기타정보/신규 "권리변동·행정" 섹션.
- **BC-8 (P2) R&D·표준 섹션**: `rnd?: {taskNo,dept,project,task,institute,period,contribution}[]`, `standard?: {org,numbers,techName,declarants,nationalities,date}` → 기타정보 하위 "국가R&D"·"표준특허".
- **BC-9 (P2) 서지 보강**: `finalDisposal?`, `claimCount?`, `originalApplication?`, `internationalApplication?` → 서지사항 행 추가.
- **BC-10 (P3) 기타**: 출원구분·번역문제출일·지정국·권리이전 이력·실시권등록일·도면 메타·서열목록.

> 시드/타입은 **KR+공통 기준**으로 먼저 채우고, 국가고유(§5)는 해당 국가 문헌일 때만 조건 렌더.

## 5. 국가 고유 필드 (JP/US/EP) — 후순위 요약

- JP: E_ABSTRACT(영문요약)·FI·FTERM·TEMA·PRIR(우선권)·CTLTR(인용)·custom(공보판/대리인구분/도면 figures/cited_documents). → 분류(FI/FTERM/TEMA), 인용(CTLTR)만 P2, 나머지 P3.
- US: CPC·UPC·REL_APPL(관련출원)·CTLTR/CTLTR_ETC(인용/기타인용)·SEARCH(검색항목)·custom(series_code·provisional·cited_documents). → UPC 분류, 관련출원, 인용 P2.
- EP: EPC·DSGN(지정국)·custom(출원인 정리번호·언어·bulletin). → 지정국·EPC P2/P3.

각국 고유는 §4 구현 시 `country`에 따라 조건 렌더하는 확장 포인트만 열어 두고, 실제 채움은 KR+공통 완료 후 진행.

## 6. 제외 항목 (노출 부적합)

내부·수집 메타: `id`, 각 `serial_number`, `patent_image.scrape_site/key_name`(→URL 변환 후 사용), `index_duplicate_serial_number`, `standard_publication_number`(표준내부), KPA 내부 코드류. → 화면 미노출, 필요 시 디버그/관리자만.
