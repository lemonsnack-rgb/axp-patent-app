당신은 이 저장소의 **기능정의서**(`docs/기능정의서.md` 단일 문서)를 UI(목업) 변경에 맞춰 동기화하는 작업을 한다.
"UI를 수정했으니 문서를 따라 갱신"하는 것이 목적이다. 아래 절차와 규칙을 **정확히** 따른다.

> **원칙 — 단일 문서 · 항상 전체 갱신**: 기능정의서는 **하나의 문서**다. 화면(검색어 입력·결과 목록·**특허 상세페이지**·공통)을 **별도 문서로 쪼개지 않는다.** 동기화 시 변경분만 부분 append 하지 말고, **문서 전체를 훑어 모든 화면의 통합표·카탈로그·data-spec 정합을 함께 갱신**한다.

## 동기화 대상 (3곳을 일치시킨다)
1. `docs/기능정의서.md` — 「메뉴·화면별 기능명세」의 **구성요소별 통합표**(각 화면 ② 표)와 공통·횡단 표
2. `src/features/spec-overlay/specCatalog.ts` — 명세 모드 오버레이가 쓰는 카탈로그(문서 표와 1:1)
3. 각 화면 컴포넌트의 `data-spec="<ID>"` 속성 — 실제 UI 부착 지점

## 절차
1. **변경 파악**: `git diff`(작업트리)와 최근 커밋을 확인해, 아래 파일에서 **UI 구성요소·액션의 추가/삭제/변경**을 식별한다.
   - `src/views/PatentInput.tsx`, `src/views/PatentResults.tsx`, `src/views/PaperInput.tsx`, `src/views/PaperResults.tsx`
   - `src/components/PatentDetail.tsx` — **특허 상세페이지(`PAT-DET-*`, 오버레이·전체보기 공용)도 대상**
   - `src/features/spec-overlay/specCatalog.ts`, 그리고 `data-spec`가 붙은 요소들
   - $ARGUMENTS 가 주어지면 그 화면을 우선 대상으로 하되, **최종적으로 전체 문서 정합을 점검**한다.
   - **논문 상세페이지만 보류** — 수집 DB 미설계로 필드 확정 불가(건드리지 않음).
2. **무엇이 바뀌었는지 결정**: 새 구성요소/액션이 생겼나? 삭제됐나? 표시 정보·상호작용·사전조건·사후결과가 달라졌나? 코드(실제 렌더)를 근거로 판단한다(추측 금지).
3. **세 곳을 갱신**한다. 아래 "작성 규칙"을 지킨다.
   - 문서 표: 해당 화면의 통합표 행을 갱신/추가/삭제
   - 카탈로그: 문서 표와 동일하게 항목 갱신(id, screen, component, display, interaction, precondition, postcondition)
   - 컴포넌트: 새 요소엔 `data-spec` 부착, 삭제된 요소는 제거
4. **정합 검사**: 모든 `data-spec` ID가 카탈로그·문서에 존재하는지, 반대로 문서/카탈로그 ID가 코드에 부착됐는지 대조하고 불일치를 보고한다.
5. **검증**: `npm run build`(타입체크 포함) 통과를 확인한다.
6. **요약 보고**: 추가/변경/삭제한 ID와 그 이유를 표로 보고한다(리뷰용). **커밋·푸시·배포는 사용자가 지시할 때만** 한다.

## 작성 규칙 (문서와 동일 규약 — 반드시 준수)
- **범위**: 특허/논문의 검색어 입력·결과 목록·**특허 상세페이지**·공통·횡단. 논문 상세페이지만 보류(제외).
- **UI에 실제 렌더되는 것만 기재(창작 금지)**. 코드로 확인되지 않는 동작·필드는 쓰지 않는다. 미구현·미확정은 `⚠ 미구현 / 연동 시 정의`로 표기.
- **목업 단계 관점**: 이용자의 **이용 흐름**과 **정보 표시 기준** 중심. 구현 방안(엔드포인트·필드 배선)은 "④ 정보 출처(연동 시 참고)"에만 후보로 둔다.
- **하이브리드 입도**: **영역**(10단위 ID: 010·020…) + 한 영역에 **사전조건/사후결과가 다른 동작이 여럿이면** 1단위로 **액션** 분리(예: `041 검색 실행`/`042 초기화`, 표에서 `└` 접두). **동일 동작(예: 국가 칩 여러 개)은 나누지 않는다.**
- **ID 채번**: `도메인(PAT/PAP/COM)-화면(INP/LST/DET)-순번` (DET=상세페이지). **기존 ID는 유지**하고, 새 항목은 빈 순번을 이어서 부여. 삭제 시 번호를 재사용하지 않는다.
- **사전조건/사후결과 정의**: 사전조건=그 기능을 쓰기 위해 먼저 충족돼야 하는 상태·입력, 사후결과=쓴 뒤 발생하는 일·상태 변화.
- **표기 금지 사항**: 자료 A/B/C 같은 코드네임 금지(실제 이름 사용: bigtext 매뉴얼/동작방식 개발노트/prior-art-search/ck-patent/프로토타입). `§` 내부번호 참조 금지(이름 기반). `검색-NN` 요구사항 코드 금지(문장으로 서술). **마크다운 링크·임의 URL 생성 금지**(문서명은 평문 백틱).
- **data-spec 부착 방식**: 영역 컨테이너 + 개별 액션 버튼 모두에 부착(영역 여백 클릭=영역 개요, 버튼 클릭=액션 명세). `@muhayu/axp-ui`의 `Button`은 `...props`를 DOM에 전달하므로 `data-spec`를 직접 붙일 수 있다.

## 주의
- 이 명령은 AI가 코드 의미를 읽어 문서를 갱신하는 **반자동** 절차다. 결과는 **사용자 리뷰가 필요**하다 — 확신이 없는 항목은 `⚠ 확인 필요`로 표기해 보고한다.
- 문서의 「배경 및 참고자료」는 UI 변경과 무관하면 건드리지 않는다. 「부록 — 논문 상세페이지 보류」는 논문 상세가 착수될 때만 갱신한다.
